## Anti-Patterns Verdict

**Does this look AI-generated?** Borderline. The CLI panel itself avoids the worst SaaS-cream tropes, but it lands in a very common "integration logo grid" pattern. The specific mistakes below are what push it toward generic.

**LLM assessment:** The panel is a competent, compact integration matrix, but it does not fully own the "Guard Rail Terminal" voice. The biggest tell is the hardcoded orange `#E58339` primary button in the adjacent quickstart callout — it is not in the committed palette, it is not a status color, and it reads as a leftover template accent. Inside the panel, using Guard Green on card hover is a semantic error: green is reserved for "allowed/protected" states, yet it is applied as decoration to static compatibility cards. These two choices make the surface feel assembled rather than designed.

## Overall Impression

The CLI panel is functional and readable, but it is playing it safe. It communicates "we support these tools" without proving it. The misuse of green for hover and the off-palette orange CTA are the two issues that most undermine the project's hard, technical, protective personality.

## What's Working

1. **Dense, horizontal card format.** Icon + name + vendor label is a scannable integration pattern; the compact padding and 6px radius keep it utilitarian rather than bubbly.
2. **Always-dark panel surface.** Keeping the CLI panel dark in both themes is correct per the Dual-Field Rule; it preserves terminal authority and keeps vendor logos readable.
3. **Responsive auto-fit grid.** `repeat(auto-fit, minmax(220px, 1fr))` reflows cleanly without extra breakpoints.

## Priority Issues

### [P1] Off-palette orange primary CTA breaks the color system
- **What:** The "Get started" button in `.cc-overview-callout` uses `background: #E58339`. This color is not in `DESIGN.md`, is not a status color, and is not used anywhere else in the semantic token set.
- **Why it matters:** A single unexpected warm orange on a dark teal-black page looks like a stock CTA from a different template. It undermines the restrained, status-only palette and the "blunt, technical, protective" voice.
- **Fix:** Replace with the documented dark-mode primary button: Bone Ink background (`#f3f1db`) with Void Teal text (`#091c1e`), and Guard Green Hover for the hover state (`--cc-guard-green-bright-hover`). The CSS class `.cc-overview-button-primary` should map to `--cc-btn-primary-bg` / `--cc-btn-primary-text`.
- **Suggested command:** `$impeccable colorize index.mdx custom.css`

### [P1] Guard Green used as decorative hover on static CLI cards
- **What:** `.cc-cli-card:hover` sets `border-color` to a mix of `--cc-guard-green-bright`. Green is defined in DESIGN.md as the allowed/protected status color.
- **Why it matters:** Hovering a static compatibility card with green implies "allowed" or "protected" semantically, but the card conveys no such state. This dilutes the meaning of the product's most important status signal.
- **Fix:** Use `--cc-rail-line-strong` or a subtle vendor-native tint on hover. If you want a positive accent, reserve green for cards that actually indicate active protection or verified integration.
- **Suggested command:** `$impeccable colorize index.mdx custom.css`

### [P1] Vendor meta label fails WCAG AA contrast
- **What:** `.cc-cli-meta` uses `--cc-terminal-muted` (`#818477`) at 0.75rem / 12px on `--cc-card-teal` (`#102b2d`). Estimated ratio is ~4.2:1, below the 4.5:1 requirement for small text.
- **Why it matters:** The vendor name is the secondary identifier for each tool; low-contrast meta text is hard to read and fails the project's stated AA target.
- **Fix:** Lighten `--cc-terminal-muted` slightly (e.g., to `#8f9285`) or use a darker card background (`#0d1f21`) behind meta labels.
- **Suggested command:** `$impeccable audit index.mdx custom.css`

### [P2] CLI cards are static display, not interactive
- **What:** The cards are `div`s with no `href`, no `role`, and no keyboard focus. A developer scanning supported tools has no path to per-tool installation or docs.
- **Why it matters:** Recognition is good, but the next natural action ("how do I set this up for Claude Code?") is unsupported.
- **Fix:** Convert each card to an `<a>` linking to a `/installation/<tool>` or anchor section. Add `:focus-visible` styling and ensure the whole card is keyboard-focusable.
- **Suggested command:** `$impeccable layout index.mdx custom.css`

### [P2] Card grid reads as generic integration matrix
- **What:** Seven same-sized cards in an auto-fit grid is a standard pattern; it does not feel specific to a command-safety interlock.
- **Why it matters:** The brand register demands a POV. A generic logo grid could belong to any developer tool.
- **Fix:** Consider grouping by ecosystem (OpenAI, Anthropic, Google, GitHub, etc.) or adding a one-line "status" per card (e.g., "shell wrapper" / "agent hook"). Alternatively, use rail-bordered cells instead of enclosed cards.
- **Suggested command:** `$impeccable layout index.mdx custom.css`

## Persona Red Flags

### Jordan (Confused First-Timer)
- **"Supported Coding CLI" heading is clear**, but after reading the list Jordan has no obvious next step. The cards are not links, so the section dead-ends.
- **No per-tool guidance:** Jordan does not know whether CC Safety Net works the same way for Claude Code vs. Pi vs. Kimi.
- **Orange "Get started" button** draws the eye, but its color does not match the rest of the page, making Jordan wonder if it is a broken stylesheet.

### Casey (Distracted Mobile User)
- **One-column reflow works**, but the 44×44pt icon touch target is only the logo image, not the whole card.
- **Cards are not tappable** because they are not interactive. Casey taps a card expecting a docs page and gets nothing.

### Skeptical Developer (project-specific persona)
- **Guard-green hover on static cards** reads as false signaling. A safety-tool audience will notice that green means "allowed" and wonder why it is being used decoratively.
- **Off-palette orange CTA** makes the project look less considered, which erodes trust for a tool that claims to be a hard safety boundary.

## Minor Observations

- `alt="Codex"` on the icon duplicates the visible title; acceptable but could be more specific (e.g., "Codex logo").
- Kimi Code is the only tool using a PNG (`kimi-cli.png`); the rest are SVGs. This may cause scaling or dark-mode inversion inconsistencies.
- `aria-label="Codex"` on a non-interactive `div` is likely ignored by screen readers; if the cards become links, move the label to the anchor.
- The panel heading "Supported Coding CLI" is slightly awkward — "Supported coding CLIs" or "Supported agents & CLIs" reads more naturally.

## Questions to Consider

- Would grouping tools by integration method (shell wrapper, agent hook, MCP) make the panel more useful than a flat alphabetical/vendor grid?
